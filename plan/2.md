# 代码解读提示词

你是一个资深的代码审阅专家和技术架构师，擅长快速理解复杂代码逻辑、识别性能瓶颈、发现安全隐患。

## 项目背景

**技术栈**:
- 后端: Spring Boot 3.2 + MyBatis + Spring Security (JWT)
- 前端: Vue 2 + Element UI + Vuex
- 数据库: MySQL
- 消息队列: Redis
- 异步处理: Python FastAPI + Celery Workers

**代码仓库位置**:
- 后端源码: `backend/src/main/java/com/langdong/spare/`
- 前端源码: `frontend/src/`
- Python 服务: `python-ai-service/app/`
- 数据库脚本: `sql/`

## 需要解读的代码

### 方式 1：按文件路径解读（完整文件分析）

**文件**: [完整的相对或绝对路径]

**解读需求**:
- [ ] 概要：这个文件的作用是什么？
- [ ] 入口：主要公开的方法有哪些？
- [ ] 流程：核心业务逻辑是怎样的？
- [ ] 依赖：这个类依赖了哪些外部组件？
- [ ] 隐患：是否存在潜在的性能问题、并发问题、安全漏洞？

**示例**:
```
文件: backend/src/main/java/com/langdong/spare/service/ai/AiForecastService.java

解读内容:
- 这个 Service 的核心职责
- 所有public方法的用途和参数说明
- 与其他服务的交互流程
- 可能存在的 bug 或性能问题
```

### 方式 3：按复杂方法解读（单个方法深度分析）

**方法签名**: 
```java
[类名].[方法名]([参数列表])
```

**方法位置**: [文件路径]:[行号]

**深度分析需求**:
- [ ] 这个方法的**参数**分别代表什么，约束条件是什么？
- [ ] 这个方法的**返回值**有什么业务含义？
- [ ] 内部**逻辑分支**有多少种，分别代表什么意思？
- [ ] 是否调用了其他方法，形成了什么样的**调用链**？
- [ ] 涉及数据库操作时，**事务语义**是什么？
- [ ] 是否存在**并发问题**或**性能瓶颈**？
- [ ] **异常处理**是否完善？

**示例**:
```
方法: AiForecastService.mapFromAsyncRow(Map row, String defaultMonth)
位置: backend/.../AiForecastService.java:324

请解释:
1. Map row 中包含哪些关键字段？
2. defaultMonth 的含义和职责？
3. 返回 List<AiForecastResult> 代表什么？
4. 内部的月份扩展逻辑（monthly_detail 到多行记录）如何工作？
5. 是否存在数据一致性风险？
```

### 方式 4：按架构设计解读（系统设计分析）

**架构主题**: [选择一个主题]
- [ ] AI 需求预测模块的总体架构
- [ ] 异步任务处理的设计模式（Job → Queue → Worker → Callback）
- [ ] 库存补货建议的生成逻辑
- [ ] 前后端的数据流向与同步机制

**分析维度**:
- [ ] 整体设计思路（流程图 / 时序图）
- [ ] 关键实体和它们的关系（ER 图）
- [ ] 核心服务层的职责划分
- [ ] 跨服务/跨进程的通信方式
- [ ] 数据一致性如何保证？
- [ ] 可扩展性和性能瓶颈在哪里？
- [ ] 有没有可以优化的地方？

**示例**:
```
架构主题: AI 异步任务与补货建议的联动流程

请分析:
- 从 UI 提交任务到最终生成补货建议，涉及哪些服务和数据库表？
- 如何确保 Python Worker 的结果被准确回调到 Java 后端？
- 每一步的数据格式和验证规则是什么？
- 是否存在数据丢失或不一致的风险？
```

### 方式 5：按性能/安全解读（专题分析）

**专题类型**: 
- [ ] 性能问题：哪些代码可能是性能瓶颈？
- [ ] 并发问题：是否存在线程安全隐患？
- [ ] 安全漏洞：input validation、SQL injection、XSS 等？
- [ ] 内存泄漏：是否有资源未释放？
- [ ] 代码质量：是否有 code smell 或技术债？

**扫描范围**:
- [ ] 指定的文件路径
- [ ] 指定的方法名称
- [ ] 指定的功能模块

**分析输出**:
- 问题列表（严重程度：CRITICAL / HIGH / MEDIUM / LOW）
- 根本原因
- 修复建议
- 优先级排序

**示例**:
```
专题: 库存计算模块的性能和并发问题

扫描文件: 
- StockCalculationService.java
- StockThresholdService.java
- ReorderSuggestService.java

请识别:
1. 是否存在 N+1 查询问题？
2. 是否存在数据库锁竞争？
3. 是否有缓存机制？缓存策略是否合理？
4. 批量操作时是否存在超时风险？
```

## 🔍 解读时的关键问题

无论选择哪种方式，都可以追问这些问题：

**对于 Java 代码**:
- [ ] @Transactional 的事务边界是什么？
- [ ] 是否使用了 ThreadLocal 或并发工具？
- [ ] SQL 是如何动态生成的（MixBatis XML 如何对应)?
- [ ] 异常处理中是否隐藏了真实错误？
- [ ] 是否存在资源泄漏（DB Connection、File Handle 等）？

**对于 SQL**:
- [ ] 索引是否合理？有没有 LIKE '%' 这样的全表扫描？
- [ ] JOIN 的表是否过多？
- [ ] 是否使用了 COUNT(*) 作为分页依据？

**对于前端代码**:
- [ ] 是否存在内存泄漏（event listener 未移除）？
- [ ] API 调用是否有错误处理？
- [ ] 状态管理（Vuex）是否清晰合理？
- [ ] 组件生命周期是否正确？
