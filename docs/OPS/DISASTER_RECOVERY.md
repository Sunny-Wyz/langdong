# 灾备与恢复方案（DR）

最后更新: 2026-03-26

---

## 1. DR 目标

建议目标:
- RPO <= 24 小时
- RTO <= 4 小时

定义:
- RPO: 可接受的数据丢失窗口
- RTO: 可接受的业务恢复时长

---

## 2. 备份策略

## 2.1 数据库备份

- 全量备份: 每日 1 次
- 备份保留: 30 天
- 至少保留 1 份异地副本

示例命令:
```bash
mysqldump -u admin -p spare_db \
  --single-transaction --routines --triggers \
  > /backup/spare_db_$(date +%Y%m%d).sql
```

## 2.2 应用与配置

- 后端 jar: 每次发布保留最近 5 版
- 前端 dist: 每次发布归档
- 配置文件: 使用安全仓库存放，禁止明文入库

---

## 3. 恢复流程

## 3.1 数据库恢复

1. 选择恢复点
2. 新建恢复库或直接恢复目标库
3. 校验核心表与关键业务数据

```bash
mysql -u admin -p spare_db < /backup/spare_db_YYYYMMDD.sql
```

校验 SQL:
```sql
SELECT COUNT(*) FROM spare_part;
SELECT COUNT(*) FROM biz_requisition;
SELECT COUNT(*) FROM ai_forecast_result;
```

## 3.2 应用恢复

1. 部署最近稳定版本后端
2. 部署最近稳定版本前端
3. 恢复 Nginx 代理配置
4. 执行健康检查与业务冒烟

---

## 4. 故障分级与升级

| 等级 | 说明 | 响应时限 |
|---|---|---|
| P1 | 生产不可用/核心业务中断 | 15 分钟内 |
| P2 | 关键功能受影响 | 30 分钟内 |
| P3 | 局部功能异常 | 2 小时内 |

升级路径:
1. 值班工程师
2. 模块负责人
3. 技术负责人
4. 管理层通知

---

## 5. 演练机制

- 频率: 每季度至少 1 次
- 演练范围: 备份恢复 + 应用回切 + 冒烟验证
- 产出: 演练记录、耗时、问题清单、改进项

演练记录模板:
1. 演练时间
2. 触发场景
3. 恢复步骤
4. 实际 RTO/RPO
5. 偏差与原因
6. 改进行动

---

## 6. 成功判定

恢复完成后，需满足:
1. 登录成功
2. 核心菜单可访问
3. 领用/采购/报表接口可返回
4. 数据库关键表行数合理
5. 监控告警恢复正常
